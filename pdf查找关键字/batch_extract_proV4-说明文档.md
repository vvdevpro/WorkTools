# 📄 自动化 PDF 智能提取工具 — 使用说明文档

## 📋 概述

**自动化 PDF 智能提取工具 v5.1**（`batch_extract_proV4.py`）是一款基于 OCR 技术的 PDF 关键字智能提取工具。它利用 **RapidOCR** 对 PDF 文件进行逐页 OCR 文字识别，并按照用户自定义的关键字规则，自动提取匹配的文本内容。

### 核心特性

1. **⚡ 极速并发** — 基于进程池的模型池化技术，每个进程只加载一次 AI 模型，避免重复加载的资源浪费
2. **⚖️ 负载均衡** — 采用"页级（Page-level）"任务切分，防止超大单文件阻塞进程，进度条更精准
3. **🛡️ 健壮容错** — 三级异常捕获（进程、文件、页面），确保残缺或损坏的 PDF 不会导致整个任务崩溃

## 🚀 快速开始

### 1️⃣ 环境要求

- **Python 3.8+**
- 依赖库列表：

| 依赖 | 用途 |
|------|------|
| `rapidocr-onnxruntime` | OCR 文字识别引擎 |
| `PyMuPDF`（`fitz`） | PDF 文件解析与页面渲染 |
| `psutil` | 进程优先级管理（防止前台卡顿） |

安装依赖：

```bash
pip install rapidocr-onnxruntime PyMuPDF psutil
```

### 2️⃣ 运行方式

将 `batch_extract_proV4.py` 与要提取的 PDF 文件放在 **同一文件夹** 下，直接运行：

```bash
python batch_extract_proV4.py
```

或双击 `.py` 文件（需已关联 Python 解释器）。

### 3️⃣ 交互流程

启动后程序进入交互式命令行界面，流程如下：

```
  启动程序
     │
     ▼
  👉 请输入提取规则
     │
     ▼
  👉 请输入并发进程数
     │
     ▼
   自动扫描当前目录下所有 PDF
     │
     ▼
   并发 OCR 提取（显示进度条）
     │
     ▼
   生成提取结果报告（TXT）
     │
     ▼
   程序退出
```

## 🔍 关键字规则语法

### 规则说明

| 语法 | 含义 | 示例 |
|------|------|------|
| **空格分隔** | 不同的关键字组，**分别独立搜索** | `收款 支付` → 搜索"收款"或"支付" |
| **`+` 号组合** | 多个关键字必须**在同一行同时出现** | `支付+%` → 搜索同时包含"支付"和"%"的行 |
| **混合使用** | 空格分隔规则组，组内用 `+` 组合 | `收款 支付+%` → 搜索"收款" 或 同时包含"支付和%"的行 |

### 示例详解

输入：`收款 支付+%`

解析为两条规则：
- **规则 1**（优先级高）：必须同时包含 `支付` + `%`
- **规则 2**（优先级低）：包含 `收款`

程序会按 **规则复杂度排序**（组合关键字越多优先级越高），每行文本只匹配优先级最高的那条规则。

### 中文兼容

支持中文加号 `＋`，会自动转换为英文 `+` 处理。

## ⚙️ 并发配置

### 自动检测

程序启动时会自动检测本机 CPU 逻辑线程数，并将最大并发进程数 **硬性锁定为逻辑线程数的一半**（至少为 1），以防止电脑卡顿。

### 用户配置

运行时会提示输入并发进程数：

```
👉 请输入想调用的并发进程数 (范围 1~N，直接按回车默认使用 N):
```

- **直接回车**：使用默认最优配置（CPU 线程数的一半）
- **输入数字**：范围为 `1 ~ CPU线程数/2`，超出上限会自动拦截并调整

### 进程优先级

每个子进程启动后会自动降低系统优先级（Windows 上设置为 `BELOW_NORMAL_PRIORITY_CLASS`），确保前台操作流畅。

## 🧬 核心技术原理

### 1. 进程池模型池化

```
主进程
 ├── 创建 ProcessPoolExecutor
 ├── initializer=init_worker   ← 每个子进程初始化时只加载一次 OCR 模型
 ├── 提交任务（每页一个独立任务）
 │
 ├── 子进程 1 ─── OCR 模型（全局变量）
 ├── 子进程 2 ─── OCR 模型（全局变量）
 ├── 子进程 3 ─── OCR 模型（全局变量）
 └── ...
```

使用 `concurrent.futures.ProcessPoolExecutor` 的 `initializer` 参数，确保 **每个子进程在整个生命周期内只加载一次 RapidOCR 模型**，避免反复加载造成的性能浪费。

### 2. 页级任务切分

```
PDF 文件 A（10 页） → 10 个独立任务
PDF 文件 B（200 页）→ 200 个独立任务
                      ↓
           全部混入同一个任务池
                      ↓
        先完成的页先返回，进度条实时更新
```

- 防止超大 PDF 文件阻塞整个队列
- 进度条粒度为"页"，更精准反映真实进度
- 天然负载均衡

### 3. 三级异常容错

| 层级 | 异常场景 | 处理策略 |
|------|----------|----------|
| **进程级** | 子进程崩溃 | `future.result()` 捕获异常，记录日志，不影响其他进程 |
| **文件级** | PDF 损坏/加密/无法打开 | 预扫描时跳过，记录错误信息 |
| **页级** | 单页渲染失败 / OCR 异常 | 捕获异常，返回空结果 + 错误信息，继续处理下一页 |

## 📂 输出结构

### 输出目录

运行完成后，会在程序所在目录下创建一个名为 `提取结果_YYYYMMDD_HHMMSS/` 的文件夹。

```
当前目录/
├── batch_extract_proV4.py        ← 主程序
├── 文档1.pdf
├── 文档2.pdf
│
└── 提取结果_20260421_143000/     ← 自动生成的输出目录
    ├── 文档1_提取结果.txt
    ├── 文档2_提取结果.txt
    └── ...
```

### 结果文件格式

每个 PDF 文件对应一个 `_提取结果.txt` 文件，格式如下：

```
--------------------------------------------------
📄 【来源】 第 3 页
🎯 【命中规则】 支付 + %
📝 【条款原文】
（此处为 OCR 识别出的原始文本行）
--------------------------------------------------
```

### 空结果

如果某 PDF 文件中没有匹配任何规则，结果文件内容为：

```
未发现匹配当前规则的内容。
```

## ⚠️ 注意事项

1. **首次运行**：第一次运行时会自动加载 OCR 模型，耗时约 5~30 秒（视 CPU 性能而定），请耐心等待
2. **GPU 支持**：`RapidOCR` 默认使用 CPU 推理，如需 GPU 加速请自行配置 `onnxruntime-gpu`
3. **PDF 加密**：加密的 PDF 文件会被自动跳过，请先解密后再使用
4. **字体依赖**：OCR 识别效果取决于 PDF 中文字清晰度，扫描件效果优于截图转 PDF
5. **文件编码**：结果 TXT 文件统一使用 UTF-8 编码
6. **路径名**：支持中文路径和文件名
7. **内存占用**：每个子进程约占用 200~500MB 内存，请确保系统有足够内存（建议 8GB+）
8. **并发数**：不建议超过 CPU 逻辑线程数的一半，否则可能导致系统响应变慢

## 🔧 常见问题

### Q: 结果文件中没有内容或内容不完整？

- 检查关键词是否匹配 OCR 识别结果（注意中文全角/半角、大小写）
- 尝试降低 PDF 渲染缩放倍数（修改 `Matrix(2, 2)` 为 `Matrix(3, 3)` 以获得更清晰的图像）
- 确认 PDF 文件未加密且内容为可识别的文字（非纯图片扫描件）

### Q: 程序启动很慢？

- 首次启动需要加载 OCR 模型文件，属于正常现象
- 后续每个子进程也需要独立加载一次模型（因为有进程隔离）

### Q: 如何处理大批量文件？

- 建议分批处理，每批不超过 100 个文件
- 根据文件的实际页数调整并发数，页数少可适当增加并发
